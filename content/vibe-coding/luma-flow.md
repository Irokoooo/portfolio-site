## 需求觉察

结合我自身关注国际经贸时事的习惯，以及近期自学法语和进阶英语的切实痛点，我发现市面上的语言学习 App 要么语料极其老旧，要么收费混乱且系统臃肿。当我尝试直接阅读新鲜的实时外刊时，往往会面临生词壁垒；而如果想就新闻内容与 AI 展开深度讨论，频繁在新闻网页和 AI 对话框之间复制粘贴，又会严重打断心流。因此，我决定打造一个工具，其底层逻辑是“动态分级与语境融合”：应用能够抓取实时新闻，根据用户当前的语言水平自动下调或拔高词汇难度，并支持在同一页面内无缝展开基于新闻情景的口语对练，彻底消除切换 App 的摩擦力。

## 痛点与挑战

在这个多模态应用的开发中，得益于 Google AI Studio 完善的生态体系，新闻源和语音（TTS/STT）接口的调通极其顺畅。真正的逻辑瓶颈卡在交互美化与项目部署上。
首先，在兼顾多端适配时，由于缺乏全局规划，AI 在优化移动端 UI 时经常会误改甚至彻底弄崩已有的电脑端页面排版。其次，项目在 Google AI Studio 沙盒内成型后，一旦同步到 GitHub 准备独立部署，就面临官方 API 额度极低且调用成本高昂的现实。这意味着我必须脱离沙盒，在本地代码编辑器中重写后端逻辑，将底层的模型服务替换为性价比更高的私有 Token 接口。

## 落地解法

为了跨越上述障碍，我摸索出了一套“双引擎”工作流：

1. 需求翻译与 Plan Mode 编排：
我不再直接对写代码的 AI 提需求。我先将大白话的痛点喂给另一个逻辑能力强的 AI（如 Claude），让其将需求“翻译”成带有专业术语的前后端开发指令。拿到结构化文档后，再进入代码 Agent 的 Plan Mode，先锁死整体骨架，再逐一填入血肉组件。

2. 底层隔离解决多端冲突：
针对多端适配频频报错的问题，我在项目核心的 skill.md（系统指令）中，强制要求 AI 从一开始就将 PC 端与移动端的 CSS/逻辑代码进行物理分隔，以此框定 AI 的修改范围，避免了“牵一发而动全身”的崩溃。

3. 节点回溯与 API 平替：
在本地 VS Code 重构接口以接入低成本 API 时，我严格要求 AI 每改动一个组件就进行一次版本记录。通过极细颗粒度的版本溯源，我成功在不破坏原有“听、说、读、记”业务流的前提下，完成了底层通信接口的剥离与替换。

## 迭代心得

从初期的磕磕绊绊到如今 Luma-Flow 的全链路跑通，我对 Vibe Coding 的工作流已经驾轻就熟。我深刻意识到，当底层代码生成的门槛无限趋近于零时，作为产品架构师的“护城河”便不再是技术实现，而是角色认知的转换：必须将自己定义为 Vibe Coding 的“消费者”而非单纯的“生产者”。只有极致地代入真实用户的痛点，不断追问“我自己究竟愿意为怎样的服务买单”，才能将一个自嗨的 Demo 真正转化为逻辑严密、具备商业落地价值的产品。
