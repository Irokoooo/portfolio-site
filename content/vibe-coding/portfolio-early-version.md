## Project Overview

作品集初版诞生于 2025 年暑期，当时正在申请函数猫（FuncCat）前端产品实习。彼时对 React 生态的了解停留在"听说过"阶段，完全依靠 AI Prompt 驱动从零搭出了第一个个人站点原型，是非常典型的 Early Vibe Coding 风格——功能驱动，视觉粗糙，但跑起来了。

---

## 需求觉察（Motivation）

时间压力很真实：实习投递窗口只有两周，简历上空有经历描述，没有任何"可点进去看"的东西。当时的判断是，前端产品岗最看重"能不能实际落地"，有一个活的、能跑的页面，远比一张充满文字的 PDF 更直接。

于是在 2025 年 7 月，我用 Claude + Cursor 在 3 天内搭出了第一版。目标非常朴素：把自己的经历、项目和联系方式塞进去，有个域名能打开就行。

---

## 痛点与挑战（Pain Points）

### 初版"丑"在哪里

- **配色是事故现场**：整页深蓝配亮橙，高对比度的科技感配色在浅色内容页上完全水土不服
- **字体层级缺失**：标题、正文、标签全是同一字号同一字重，页面读起来像在看说明书
- **信息密度失控**：想把所有经历全塞进首页，导致每个板块都挤在一起，白色空间几乎为零
- **组件复用率为零**：每个卡片都是独立堆砌的 HTML，没有任何抽象，改一个颜色要手动改十几处
- **没有响应式**：移动端打开是一片混乱，文字溢出容器、布局错位

### Vibe Coding 的局限性

作为一个商科生跳进前端，最大的盲区是"不知道自己不知道什么"。AI 帮我生成了很多看起来"能跑"的代码，但这些代码是堆砌的，不是设计的：
- 组件没有职责边界，一个文件里塞了七八个不相关的功能
- CSS 是一大坨内联 style，没有任何设计 Token 的概念
- 状态管理靠 prop drilling 传了三四层，改一个联动需要追踪半天

## 落地解法（Implementation）

### 交付给函数猫的版本

页面最终在投递截止前两天上线，包含：
- 个人简介页（About）：带照片、简短自我介绍、技能标签
- 项目展示页（Projects）：4 个卡片，各有封面图和简短描述
- 联系方式：邮箱 + GitHub + LinkedIn 图标行

用 Vercel 部署，生成了独立域名，直接贴在简历"个人主页"一栏。

函数猫的面试官看了之后说了一句话，我印象非常深刻：**"页面跑起来了，但代码质量我没法评价"**。

这句话是我后来决定彻底重写作品集的直接原因。

## 迭代心得（Iteration Notes）

### Before → After 的核心差距

| 维度 | 初版（函数猫版本） | 现版 |
|------|---------|---------|
| 技术栈 | React + Vite + 大量内联 style | Next.js 14 App Router + TypeScript + Tailwind |
| 设计系统 | 无，全靠即兴 | Tailwind 自定义 Token（seed-shadow / cream-pour 等） |
| 字体 | 系统默认 | Instrument Serif + Geist Sans（衬线/无衬线分层） |
| 配色 | 随机堆叠 | 植物学暖调羊皮纸系（有理论依据的低饱和度系统） |
| 组件质量 | 无复用，无类型 | TypeScript 接口定义，组件有清晰 Props |
| 内容管理 | 硬编码在 JSX 里 | MDX + index.ts 注册表，内容与展示分离 |
| 交互 | 无动画 | Framer Motion 全局系统（入场 / 悬浮 / 页面切换）|
| 部署 | Vercel 裸部署 | 同 Vercel，但有 CI 脚本与类型检查 |

### 这段经历教会我什么

Vibe Coding 的核心价值不是"写出好代码"，而是**"快速把想法变成可见的东西"**。初版的价值恰恰在于它跑起来了——它让我拿到了面试机会，也让我在面试官的那句话里，第一次真实感受到"能用"和"好用"之间的距离。

这个早期版本是我 Vibe Coding 学习曲线的起点。它很丑，但它是真实的。
